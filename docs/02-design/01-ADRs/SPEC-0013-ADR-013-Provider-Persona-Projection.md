# ADR-013: Provider-Specific Persona Projection

**SPEC ID**: SPEC-0013
**Status**: APPROVED (Sprint 23 Research)
**Date**: 2026-03-07
**Deciders**: [@cto]
**Tag**: `adr-013-provider-persona`
**Depends on**: ADR-010 (Bridge), ADR-011 (SOUL-Aware Launch)

---

## Context

MTClaw's bridge manages AI coding sessions across multiple providers (Claude Code, Cursor, Codex CLI, Gemini CLI). Sprint 18 implemented SOUL persona injection for Claude Code via `--agent` / `--append-system-prompt-file`. The question: how do other providers handle persona injection, and can we create a unified abstraction?

### Why This Matters

Each provider has a fundamentally different mechanism for persona injection, knowledge loading, and rule enforcement. A naive `InjectSOUL(provider, content)` abstraction would either:
- Be too generic to leverage provider-specific features (lowest common denominator)
- Be too Claude Code-specific to work correctly with other providers

This ADR documents the actual capabilities of each provider and defines per-provider "projection contracts" — how MTClaw's SOUL concept maps to each provider's native format.

## Per-Provider Projection Contracts

### Claude Code (FULL SUPPORT — Production)

**Status**: Production (Sprint 13-21, 150+ tests)

**SOUL Projection**:
```
SOUL-{role}.md → .claude/agents/{role}.md (Strategy A)
                → --append-system-prompt-file temp.md (Strategy B)
                → bare launch (Strategy C)
```

**Knowledge Projection**:
```
SDLC Framework → .claude/skills/sdlc-framework/SKILL.md
Project context → CLAUDE.md (generated by mtclaw bridge init-project)
Turn context → sendKeys prefix (Sprint 20B)
```

**Rule Projection**:
```
ToolPolicy → --allowedTools (UX convenience)
RiskMode → --dangerously-skip-permissions + bridge Layer 1
Cost cap → --max-budget-usd (future)
Turn limit → --max-turns (future)
```

**Gap**: None — full parity with MTClaw's governance model.

### Cursor (PARTIAL — File-Only, No CLI)

**Status**: POC (Sprint 23)

**Critical limitation**: Cursor has no headless CLI mode. It runs as an IDE. MTClaw cannot launch Cursor sessions via tmux. The only integration path is **file generation** — MTClaw generates Cursor-compatible config files that users open in their IDE.

**SOUL Projection**:
```
SOUL-{role}.md → .cursor/rules/{role}.mdc
```

Cursor uses `.mdc` (Markdown Cursor) files with YAML frontmatter that supports glob patterns for file matching. The frontmatter schema:

```yaml
---
description: SOUL persona for coder role
globs: ["**/*.go", "**/*.ts"]  # Optional: file-type scoping
alwaysApply: true              # Load for every conversation
---
[SOUL body content]
```

**Knowledge Projection**:
```
SDLC Framework → .cursor/rules/sdlc-framework.mdc (alwaysApply: true)
Project context → .cursorrules (legacy) or .cursor/rules/project-context.mdc
```

**Rule Projection**:
```
ToolPolicy → NOT SUPPORTED (Cursor manages its own tool access)
RiskMode → NOT SUPPORTED (Cursor IDE controls permissions)
```

**Gap**: No runtime session management. No hooks. No permission control. File generation only.

### Codex CLI (PARTIAL — CLI, No Hooks)

**Status**: Research only (Sprint 23)

**SOUL Projection**:
```
SOUL-{role}.md → AGENTS.md section (single file, multiple sections)
              → --system-prompt-file temp.md (flag-based)
```

Codex CLI uses `AGENTS.md` as its project-level instruction file (similar to CLAUDE.md). Multiple agent definitions go in sections within the same file.

**Knowledge Projection**:
```
SDLC Framework → AGENTS.md section or separate markdown file via --system-prompt-file
Project context → AGENTS.md (auto-loaded)
```

**Rule Projection**:
```
ToolPolicy → --full-auto (skip approvals) or interactive approval
RiskMode → Limited: full-auto or manual. No fine-grained policy.
```

**Gap**: No permission hooks (cannot implement bridge Layer 1). No tool allowlist. No lifecycle events. Would require polling-based monitoring.

### Gemini CLI (MINIMAL — File-Only, Limited Control)

**Status**: Research only (Sprint 23)

**SOUL Projection**:
```
SOUL-{role}.md → GEMINI.md section (single file)
```

Gemini CLI uses `GEMINI.md` as the sole project instruction file.

**Knowledge Projection**:
```
SDLC Framework → GEMINI.md section
Project context → GEMINI.md (auto-loaded)
```

**Rule Projection**:
```
ToolPolicy → NOT SUPPORTED (Gemini CLI manages its own approvals)
RiskMode → NOT SUPPORTED
```

**Gap**: No hooks. No persona switching. No tool control. Minimal integration surface.

## Decision

**DECISION: Per-provider projection, NOT unified abstraction (2026-03-07)**

### Architecture

```
ProviderAdapter (interface — D9)
  ├── ClaudeCodeAdapter      — FULL: launch, hooks, persona, skills, governance
  ├── CursorProjectionAdapter — FILE-ONLY: generate .cursor/rules/*.mdc
  ├── StubAdapter(codex-cli)  — STUB: research complete, no production adapter
  └── StubAdapter(gemini-cli) — STUB: research complete, no production adapter
```

### Key Decisions

1. **No unified `InjectSOUL()` abstraction** — Each provider's persona mechanism is too different. Claude Code has agents + skills + hooks. Cursor has .mdc rules. Codex has AGENTS.md. Forcing a common interface would hide critical capability gaps.

2. **Cursor gets a projection adapter, not a session adapter** — Cursor cannot be launched headless. The `CursorProjectionAdapter` generates files only. It does NOT implement `LaunchCommand()`, `InstallHooks()`, or `ParseStopEvent()`. These return unsupported errors.

3. **Codex CLI and Gemini CLI remain stubs** — Neither supports permission hooks (D7 Layer 0 blocker). Without hooks, sessions cannot be escalated to `interactive` mode. Building adapters without this capability would be misleading.

4. **File generation is the common denominator** — All providers support some form of project-level instruction file. MTClaw can generate these via `install-agents` with provider-specific formatters, even for providers that can't be managed as bridge sessions.

### Provider Readiness Timeline

| Provider | File Generation | Session Management | Estimated Ready |
|----------|----------------|-------------------|-----------------|
| Claude Code | NOW (Sprint 18) | NOW (Sprint 13-17) | Production |
| Cursor | NOW (Sprint 23 POC) | NEVER (no headless CLI) | File-only |
| Codex CLI | Sprint 24+ | When hooks added | TBD |
| Gemini CLI | Sprint 24+ | When hooks added | TBD |

## Implementation: CursorProjectionAdapter

### Design

The `CursorProjectionAdapter` implements `ProviderAdapter` with honest capability reporting:

```go
type CursorProjectionAdapter struct{}

func (a *CursorProjectionAdapter) Name() AgentProviderType { return AgentCursor }

func (a *CursorProjectionAdapter) LaunchCommand(_ LaunchOpts) *exec.Cmd {
    return nil // Cursor has no headless CLI
}

func (a *CursorProjectionAdapter) CapabilityProfile() ProviderCapabilities {
    return ProviderCapabilities{
        PermissionHooks:   false,
        TranscriptParsing: false,
        HookFormatVersion: 0,
    }
}
```

### File Generation

`GenerateCursorRules()` converts SOUL files to Cursor `.mdc` format:

```
Input:  docs/08-collaborate/souls/SOUL-coder.md
Output: .cursor/rules/coder.mdc
```

Format:
```yaml
---
description: SOUL persona for coder role
alwaysApply: true
---
[SOUL body content]
```

### Integration with `install-agents`

Future extension (Sprint 24+): `mtclaw bridge install-agents --provider cursor` generates `.cursor/rules/` files alongside `.claude/agents/`. Sprint 23 provides the POC adapter and generation function only.

## Consequences

### Positive
- Honest capability reporting — no false parity claims
- Each provider uses its native format (no lossy conversion)
- File generation works for ALL providers (common value)
- Clear upgrade path: when providers add hooks, promote stub → full adapter

### Negative
- No single API for "inject persona into any provider"
- Cursor users don't get session management (IDE limitation, not MTClaw limitation)
- More code paths to maintain per provider (mitigated: most are thin)

### Neutral
- Existing Claude Code adapter unchanged (Sprint 13-21 code untouched)
- `install-agents` extensibility preserved for future `--provider` flag
- Provider registry pattern (Sprint 13) already supports multiple adapters

## Unsupported Reason Codes

Each adapter reports why certain operations are unsupported:

| Code | Meaning |
|------|---------|
| `no_headless_cli` | Provider has no CLI mode (Cursor) |
| `no_permission_hooks` | Provider doesn't support hook callbacks |
| `no_transcript_parsing` | Provider doesn't output parseable transcripts |
| `file_generation_only` | Adapter can generate files but not manage sessions |

These codes are returned in error messages and logged in audit events when users attempt unsupported operations.
